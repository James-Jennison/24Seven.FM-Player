Prepare once

Make every result reproducible

Record the build and environment before touching the controls. Run only assigned account, posting, request, report, accessory, and future-delivery cases; mark a missing prerequisite Blocked rather than improvising around it.

  1. 1

    Identify the build

    Use an approved Play testing track or supplied release channel. Record the app version and code, installation source, device model, Android/API version, form factor, connection, theme, font/display scale, and relevant accessories.

  2. 2

    Take an assignment

    Record the PT case ID, station, and configuration. One report covers one case on one environment; repeat a case on another station or device only as assigned.

  3. 3

    Use safe test data

    Use only approved test accounts and synthetic dates or messages. Never disclose a real date of birth, tester identity, password, security answer, private conversation, or session value.

  4. 4

    Stop at the first unsafe boundary

    Do not send a Chat post, song request, abuse report, or external share unless that exact mutation is authorized. Do not repeat an indeterminate mutation; inspect the visible destination and report what happened.

Outcome vocabulary

Pass, notes, fail, or blocked

Pass means every observable boundary in the case occurred. Use Pass with notes when it worked but was confusing, slow, or inconsistent; Fail when an expected boundary did not occur; and Blocked when the build, account, permission, station, network control, device, or accessory was unavailable.

PT-01–PT-08 · Foundation and listening

Prove the app can arrive, play, and recover

These cases cover delivery, navigation, every station, the MediaSession, the service-owned timer, Android output routing, station data, and bounded network recovery.

PT-01 · Assigned install tester

Install, update, and first launch

  1. Install the assigned Play-delivered build or approved release APK and confirm the launcher icon, app label, version, and version code. On Android 13+, also enable themed icons and inspect the monochrome variant.
  2. Cold-launch once and exercise both notification-permission outcomes across the assigned Android 13+ devices; public playback must remain usable if permission is denied.
  3. Update an eligible earlier Alpha in place and verify local settings and protected station sessions remain available.
  4. On the designated clean-install device, uninstall and reinstall; confirm protected sessions and app-local choices are not restored from Android backup.
  5. Confirm launch has no crash, ANR, unexpected browser surface, or automatic audio start.

Pass: install and update complete under the expected identity; retained data survives only an in-place update, and a fresh install starts clean.

PT-02 · All configurations

Navigation, preferences, and exit

  1. Reach Player, Favorites, Chat, Queue, and More; confirm all five station cards are horizontally reachable.
  2. From each non-Player destination, use the persistent mini-player and return to the full Player.
  3. Choose station A, set Resume last station, select station B, then force-stop/relaunch; confirm station B opens selected without auto-playing.
  4. With station B selected, set Use current station at startup and record B as the fixed station. Select a different station C, then force-stop/relaunch; confirm station B—not the newer station C—opens selected without auto-playing.
  5. Exercise the two-Back exit flow. Verify both Keep listening and Stop and exit outcomes in separate runs.

Pass: navigation and selected state stay coherent, startup choices restore exactly, and exit never stops audio without an explicit choice.

PT-03 · Every station across the cohort

Foreground playback and station switching

  1. For the assigned station, start playback and observe Idle/Buffering/Playing or an understandable error.
  2. Confirm audible live audio, selected-station styling, title presentation, artwork behavior, format/bitrate when supplied, and matching mini-player/system metadata.
  3. Pause and resume, then move to the previous and next station, including wraparound at the ends.
  4. Switch stations while playing and confirm the prior stream and metadata do not continue as a second logical player.
  5. Stop, then start again deliberately. Repeat PT-03 until all five stations have a result.

Pass: controls and visible state agree, metadata is station-scoped, and only the selected station is audible.

PT-04 · System-media tester

Background playback and media controls

  1. Start playback, press Home, lock/unlock the device, and leave it playing for the assigned soak period.
  2. Use notification, lock-screen, Bluetooth/headset, and hardware play/pause controls where available.
  3. Tap the notification body and confirm the existing Player task opens instead of creating a duplicate task.
  4. Remove the app task from Recents while listening; verify the service/session and system media surface remain coherent.
  5. Exercise an assigned transient audio-focus interruption and a permanent interruption; verify resume occurs only when appropriate.
  6. Stop playback and confirm the system media state no longer implies that audio is active.

Pass: one MediaSession owns every control, background audio survives lifecycle changes, and foreground/system state never contradicts playback.

PT-05 · Sleep-timer tester

Sleep Timer lifecycle

  1. While playing, open Sleep Timer and exercise a preset plus valid custom 1- and 720-minute boundaries; verify invalid values cannot start a timer.
  2. Confirm the live countdown, then pause/resume, switch stations, background/lock the app, remove and reopen the task, and return; the original deadline continues.
  3. Adjust the active timer and verify it replaces the prior deadline.
  4. Cancel the timer and confirm playback continues; start another timer and use manual Stop to confirm both playback and timer clear.
  5. Run a one-minute locked-device expiry and confirm the existing player stops once, the timer clears, and any available system cancel action disappears.
  6. On the assigned reboot subcase, reboot with a timer active and confirm it fails safe instead of unexpectedly extending the deadline.

Pass: the service owns one bounded deadline that restores in the UI and never creates or controls a second player.

PT-06 · Accessory tester

Audio-output selection and route loss

  1. Confirm the Player's current output name agrees with the device actually producing media.
  2. Open Audio output, dismiss it, and confirm the same Player task remains.
  3. With assigned Bluetooth, wired/USB, hearing-aid, HDMI, automotive, or remote hardware, switch routes and verify Android System UI and the Player agree.
  4. Switch back to the device speaker, then disconnect the active accessory and observe the selected fallback route and playback state.
  5. If the system chooser is unavailable, confirm the app reports that boundary without breaking playback.

Pass: Android owns discovery/handoff, the displayed active route stays accurate, and route loss leaves a usable, truthful state. Google Cast is not claimed by this case.

PT-07 · Every station across the cohort

Queue, History, and station data

  1. Open Queue for the assigned station and verify ordered upcoming tracks plus recently played content.
  2. Check only supplied title, artist/composer, album, duration, artwork, requester, and request-message values appear; missing fields remain absent.
  3. Before community access is enabled, confirm public requester identity and messages remain hidden; after PT-14, confirm permitted attribution appears separately from track metadata.
  4. Refresh once, switch stations, and confirm old rows, artwork, requester identity, and error state do not leak into the new station.
  5. Cover SST/1980s/Adagio/Entranced extended lists and Death.FM's compact list across the cohort.

Pass: Queue and History remain bounded, ordered, capability-aware, station-scoped, and honest about unavailable data.

PT-08 · Network-control tester

Network loss, cancellation, and recovery

  1. Start live playback, disable the active connection, and wait through buffering/fallback to the explicit offline waiting state.
  2. Restore validated connectivity and verify playback prepares once and returns without another Play action.
  3. In separate runs, pause, stop, or switch stations while waiting; restoring connectivity must not restart the cancelled intent.
  4. Repeat one Wi-Fi/mobile-data handoff without a full outage.
  5. While offline, try Queue refresh, Chat refresh, account loading, Favorites, and library search. Confirm generic errors reveal no endpoint details; cached Queue data is marked stale and request actions fail closed.
  6. Restore service and confirm a deliberate refresh recovers the affected content.

Pass: recovery is bounded and intent-aware, cached state is labeled, mutations stay disabled without fresh evidence, and no private network detail reaches the UI.

PT-09–PT-19 · Accounts, participation, and support

Test protected and state-changing paths deliberately

Read-only portions may be run by any assigned tester. Posting, request, moderation-delivery, and cross-account mention checks require explicit approval and disposable test data.

PT-09 · Approved station account

Sign-in, restoration, and isolation

  1. In More, confirm all five station account states are visible without changing the listening station.
  2. Load the assigned station's native sign-in challenge, refresh the security code once, and sign in with the approved account.
  3. Force-stop and relaunch; confirm only that station restores its display identity without requesting the password again.
  4. Sign in to a second assigned station or inspect its signed-out boundary, then sign out of the first; confirm the other station is unchanged.
  5. Force-stop/relaunch again and verify the signed-out station remains cleared. If natural expiry occurs, record the explicit expired/recovery state.

Pass: credentials remain transient, protected sessions restore only for their originating station, and station-only logout remains isolated.

PT-10 · Approved account with Favorites

Favorites filtering, sorting, and availability

  1. Confirm the signed-out station-qualified gate, then sign in and load only the selected station's Favorite tracks.
  2. Filter by visible metadata and verify the result count and empty-filter state.
  3. Exercise all sorts: #, Track Name, Album, Artist, Genre, Year, Length, and Play state.
  4. On the assigned large SST list, traverse from the first to the last of approximately 1,500 tracks and keep filtering/sorting responsive.
  5. Verify an available track shows green Request Now; queued/recent tracks show red Track Recently Played and cannot be selected; stale or ambiguous data fails closed.
  6. Switch station and sign out; confirm the list and account-derived state clear without leaking.

Pass: the server-owned list is complete, responsive, sortable, station-scoped, and never enables an unsafe request.

PT-11 · Read-only request browsing

Library search and pre-submit checks

  1. While signed out, browse the library and confirm request submission remains gated.
  2. Search useful assigned queries by Title, Album, Artist, and Genre; open results and verify album/track metadata.
  3. Exercise random and least-played suggestions, plus Library order and Play state sorting.
  4. Open an available track's confirmation, verify station/track details, then Cancel and confirm no request appears in Queue.
  5. Confirm queued, recently played, cooldown, membership, limit, stale, ambiguous, and unavailable states use distinct text and do not expose an enabled action.

Pass: browsing works without mutation, Cancel is safe, and confirmation is reachable only after fresh, station-scoped eligibility.

PT-12 · Explicit mutation approval required

One-shot song request

  1. Confirm the coordinator-approved station/account is signed in, ready, outside cooldown, and allowed exactly one request.
  2. Select one green eligible track and verify the confirmation identifies the intended station and track.
  3. On SST, exercise the optional request message within its 80-character limit; on other stations confirm the SST-only field is absent.
  4. Confirm once and record the exact success, rejection, or indeterminate result. Never tap Confirm twice.
  5. Inspect Queue before any further action. Verify a received request appears once, with requester and optional message separate from metadata.
  6. If the result is indeterminate, do not retry. If a cooldown becomes visible, confirm it prevents another request.

Pass: one explicit confirmation causes at most one mutation, the UI reports uncertainty honestly, and no automatic or tester retry duplicates it.

PT-13 · SST account only

Request activity and membership evidence

  1. Confirm Request activity is unavailable while signed out and is not inherited by another station.
  2. Sign in to SST, open More → Request activity, and refresh manually.
  3. Record the explicit membership state, station-reported readiness/cooldown, and up to ten recent request summaries without inferring a tier from generic wording.
  4. Cross-check the latest authorized PT-12 request when available.
  5. Sign out and confirm the memory-only activity, readiness, and membership presentation clear.

Pass: SST evidence is explicit, read-only, bounded, manually refreshed, and cleared with the SST session.

PT-14 · Fresh-data safety profiles

Age, Terms, reveal, and legal disclosures

  1. On a clean profile, confirm Chat and requester attribution are hidden before the age screen. Use assigned synthetic dates only; verify an invalid date is rejected.
  2. On the designated underage branch, confirm community features remain unavailable while playback and non-community station data still work.
  3. On a separate clean adult branch, review Terms, Decline once, then verify Agree stays disabled until I Agree is checked.
  4. Accept Terms and confirm mature content remains hidden until separately revealed; hide it again and verify the gate returns.
  5. Relaunch and confirm only the adult/not-adult result, Terms version, and reveal choice persist—not the entered date.
  6. In More, open and scroll the Terms, privacy notice, and open-source licenses; close each and return to the same app state.

Pass: each gate is independent and persistent as documented, underage users retain listening access, and legal text is readable without disclosing a real birth date.

PT-15 · Posting requires approval

Chat reading and posting

  1. After PT-14, load the assigned station's Chat and verify bounded author, message, timestamp, and smiley-alt presentation.
  2. Refresh once, switch station, and confirm messages and errors remain station-scoped. Relaunch to confirm prior Chat history is not presented as persisted app data.
  3. While signed out, confirm reading remains available but the composer does not.
  4. With an approved account, verify the 255-character counter and that unsupported legacy characters are rejected rather than silently changed.
  5. Only when authorized, send one harmless approved message, confirm it appears once, and do not retry an indeterminate post.

Pass: Chat is gated, bounded, memory-only, correctly encoded, station-scoped, and posts only after one explicit Send.

PT-16 · Delivery requires approval

Report, block, and unblock safeguards

  1. From both Chat and request attribution, inspect the separate Report content, Report user, and Block user actions.
  2. Open a report and verify the intended station/user, bounded content preview, category, name/email, optional details, CAPTCHA, disclosure, and disabled-until-complete Send action.
  3. Cancel and reopen; confirm entered report fields were not persisted.
  4. Only the designated moderation tester may send one authorized harmless report. Record submitted, rejected/retryable, or indeterminate/non-retryable behavior and never resend an indeterminate report.
  5. Block an approved test identity; confirm its Chat messages and request attribution disappear immediately only for that station, then relaunch and confirm the local block persists.
  6. In More → Community safety, unblock it and confirm permitted content returns. Cross-check that blocked authors cannot trigger PT-17.

Pass: reporting is explicit and one-shot, form data remains transient, and local station-scoped blocks consistently filter every supported surface.

PT-17 · Two approved Chat accounts

Local Chat-mention notifications

  1. Across assigned Android 13+ devices, test notification permission allowed and denied; denial must not block Chat.
  2. Sign in, complete community gates, enable mentions for one station, relaunch, and confirm the station-scoped choice persists.
  3. Keep that station's Chat selected and allow the first snapshot to baseline without alerting for old messages.
  4. From another approved account, post an exact full display-name mention. Verify one private notification names the station and sender but contains no message text.
  5. Verify self-authored, longer embedded-name, substring, blocked-author, historical, and duplicate snapshots do not alert.
  6. Tap the notification and confirm it selects the originating station and opens native Chat without bypassing any gate; then turn the option off.

Pass: exact eligible mentions alert once with a minimal payload. The current preview observes only actively refreshing Chat; no alert while closed or away from Chat is expected yet.

PT-18 · Every station across the cohort

Trusted station pages and browser isolation

  1. For the assigned station, open both entries currently shipped under Station links: Contact and the station's VIP or RIP membership page.
  2. Confirm Android uses a secure browser Custom Tab at the expected HTTPS station host, never an in-app WebView.
  3. Press Back and verify the existing app task, selected station, navigation state, timer, and playback remain intact.
  4. If a station website requests sign-in, confirm its browser session is separate from the protected native app session.
  5. Record a clear unavailable/rejected/no-browser state rather than navigating to an unexpected host.

Pass: only the allowlisted same-station pages open, browser and app sessions remain isolated, and Back returns to the live native task.

PT-19 · Release-build support check

Privacy-safe diagnostics

  1. Create a few playback, network, and route transitions, then open More → In-app diagnostics.
  2. Review version/build type, coarse device/API, station, bounded playback/error/network/output categories, and no more than five recent enum-only transitions.
  3. Confirm there is no account identity, station/community content, title/artist, route display name, endpoint, stable device identifier, raw error, or log.
  4. Tap Copy and verify the clipboard text matches the visible preview.
  5. Tap Share, verify Android's chooser contains the reviewed text, then dismiss it without sending; confirm the app never chooses a recipient.
  6. Reopen after process recreation and confirm the app did not persist an exported report.

Pass: the visible fixed-allowlist snapshot is accurate and private, and export occurs only through an explicit user-controlled Android action.

PT-20–PT-22 · Device, accessibility, and delivery

Cover configurations that automation cannot judge

PT-20 and PT-21 are current physical/manual acceptance gates. PT-22 is published now so the future closed-app notification build has an explicit acceptance contract; it is Blocked—not failed—against the current active-Chat preview.

PT-20 · Assigned form factor

Adaptive UI, theme, and lifecycle continuity

  1. Exercise assigned compact portrait/landscape, tablet portrait/landscape, multi-window/freeform, and foldable open/tabletop/closed/reopened windows.
  2. Verify bottom navigation, rail, and two-pane Player appear at suitable widths without hiding any destination, station card, primary control, dialog action, or list content.
  3. Rotate or resize while playing and while viewing an expanded More section; confirm meaningful selected state and the existing player remain coherent.
  4. Check system bars, cutouts, keyboard, light/dark themes, and Android 16 edge-to-edge/predictive-Back behavior.
  5. On a physical foldable, run the full hinge cycle while playing and confirm task, audio, selected station, metadata, and controls survive without crash or ANR.

Pass: content reflows instead of clipping, lifecycle changes preserve one task/player, and every primary action remains visible and operable.

PT-21 · Human accessibility tester

Large text and spoken TalkBack traversal

  1. At the assigned maximum font and enlarged display settings, traverse Player, Favorites, Chat, Queue, More, account cards, dialogs, and long/empty/error lists.
  2. With TalkBack on a physical device, start from launch and reach all five destinations, station radio choices, playback, Sleep Timer, Audio output, community gates, request status/actions, and diagnostics.
  3. Listen for meaningful names, roles, selected/disabled/state announcements, focus order, pronunciation, pacing, and the Terms I Agree checkbox.
  4. Confirm green/red request states also have exact text, artwork is not a duplicate focus trap, and no actionable node is unlabeled.
  5. Repeat the assigned orientation or fold posture and confirm focus does not strand the tester or hide an action.

Pass: every task is understandable and completable through spoken feedback at enlarged settings, without relying on color, position, or sight alone.

PT-22 · Future M27.2 build only

Closed-app notification delivery

  1. Do not run this case against the current active-Chat preview. Mark it Blocked until an authorized true-push build and event source are supplied.
  2. When supplied, test permission allow/deny, foreground, background, task-removed/closed, device-idle, reboot, reinstall/token rotation, and network outage/recovery states.
  3. Across assigned station/account pairs, verify exact-name mapping, self/blocked/duplicate suppression, station-specific opt-in/out, sign-out, and minimal private lock-screen payloads.
  4. Tap an alert and verify station selection plus gated native Chat navigation; no protected session or message text may pass through a relay.
  5. Record delivery delay, duplicate count, data/battery observations, and outage behavior. Private Message delivery remains excluded until M17 is repaired and verified.

Pass: authorized events arrive once under the documented lifecycle states, respect privacy and user controls, and never depend on continuous background polling.

Structured intake

Submit one case, station, and environment per result

The public form captures the PT case ID, build, device, station, outcome, affected areas, expected and actual behavior, repeatability, reproduction steps, and redacted evidence. A clean pass is evidence. Split unrelated failures or observations into separate reports.

Critical

Safety or data exposure

Credential/session exposure, unintended private content, destructive account behavior, or a repeatable crash that prevents use. Stop testing and report immediately without attaching sensitive material.

High / Medium

Core function failure

Playback, navigation, account isolation, requests, community safeguards, delivery, or an adaptive layout cannot complete its intended job. Record exact steps and repeatability.

Low / Note

Usability or polish

The task completes, but wording, spacing, latency, spoken feedback, or consistency should improve. Select Pass with notes when the expected result still occurred.

Public report

Redact before submitting

GitHub issues are public. Never include passwords, security answers or CAPTCHA images, cookies, session values, account email addresses, real dates of birth, private messages, abuse-report content, signing material, private endpoints, full network captures, or screenshots containing personal information.

Do not repeatedly submit a song request, Chat post, or moderation report while investigating an indeterminate response. Inspect Queue or the relevant visible state once, record the outcome, and stop. Do not test closed-app push against PT-17 or Google Cast/Private Messages as if they are shipped capabilities.

  • Case ID included
  • Public metadata only
  • One environment per result
  • Redacted screenshots
  • No account secrets
  • No raw traffic captures

Record the result while it is fresh.
